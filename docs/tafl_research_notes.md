## 1. Правила вариантов (✅ верифицировано)

Общее для всех: фигуры ходят **как ладья** (любое число пустых клеток по
ортогонали, без прыжков); взятие **custodial** (зажать врага с двух
противоположных сторон своей фигурой/своей фигурой или враждебной клеткой),
срабатывает только когда **ходящий сам закрывает захват**; осаждающие ходят
первыми.

### Brandubh 7×7 ✅
- Король + **4** защитника против **8** осаждающих.
- Король **вооружён**; **углы враждебны**, **трон — НЕ враждебен**; на трон/углы
  встаёт только король; все фигуры **проходят сквозь пустой трон**, не
  останавливаясь; король **не может вернуться** на трон.
- Король побеждает, достигнув **угла**. Осаждающие — окружив короля (в этом
  рулсете **с 2 сторон**; правила Brandubh у разных авторов расходятся).
- Баланс: лучшая стартовая расстановка — **«Brandubh cross» +1.32** (cross >
  circle +1.60 > line +1.97). Рулсет «Brandubh 2» ~ **+1.53** в пользу защитника.

### Tablut 9×9 ✅
- **25 фигур**: король + **8** защитников («шведы») против **16** осаждающих
  («московиты»), «крест с засечками».
- Король **может** помогать во взятии (в рулсете Cyningstan; в части рулсетов —
  безоружный).
- **Взятие короля зависит от места**: на троне-замке — **4 стороны**; рядом с
  замком — **3**; в остальном поле — как обычная фигура (**2 стороны**).
- Король побеждает, достигнув **ЛЮБОЙ кромки** (не угла!). Ничья — повтор/нет
  хода/соглашение.
- Историческая база — Linnaeus 1732; **ошибка перевода 1811** (требовала 4
  фигуры на взятие короля вместо 2) внесла дисбаланс; оригинал **сбалансирован**.
- **Считается наиболее сбалансированным** вариантом (свежий турнир ~**50:50**).

### Fetlar 11×11 ✅
- Король + **12** защитников против **24** осаждающих.
- **Сильный король**: берётся только окружением с **4 сторон**, сам **вооружён**.
- Углы + трон — только королю, **враждебны**.
- Защитники: король → **угол** (без edge-escape). Осаждающие: взять короля.
- ⚠️ Проблема **«draw fort»** (защитник запирается в ничейную крепость) — главная
  причина появления Copenhagen.

### Copenhagen 11×11 — современный турнирный стандарт ✅
Создан **09.2012** (Nielsen, Bartley, Millar). База как Fetlar + правки баланса:
- Король **вооружён**, берётся с **4 сторон** (**3** — если рядом с троном);
  **на кромке короля взять нельзя**.
- **Shieldwall capture**: ряд из ≥2 фигур вдоль кромки берётся целиком, если
  «забрать в скобки» оба конца, и у каждого в ряду спереди враг (угол может
  заменить один конец).
- **Exit fort**: король касается кромки, может ходить, и осаждающие не могут
  разбить крепость → победа защитника.
- **Окружение («David Brown»)**: осаждающие окружили короля **и всех** защитников
  неразрывным кольцом → победа осаждающих.
- **Все повторения запрещены** (вечный повтор = поражение белых), ничья — только
  при нехватке фигур.
- Баланс: **+1.51** / «136 побед защитников на 100 осаждающих» — лёгкий перевес
  защитника.

## 2. Баланс (✅ верифицировано)

- В среднем по записям партий **осаждающие выигрывают ~9% чаще** (исторический
  «прозащитный» перекос — следствие ошибки перевода Tablut 1811).
- **Размер доски сам по себе баланс не чинит — чинит рулсет.**
- Метрика aagenielsen: `balance = победы успешного цвета / победы менее успешного`,
  ничьи = ½; **+ в пользу белых (защитников), − в пользу чёрных (осаждающих)**.
- ⚠️ **Поправка верификации:** запись вида «Brandubh +1.53 (2296/74)» — это
  `(число игр / число игроков)`, **не** счёт побед. Сам коэффициент +1.53 верен.
- Near-perfect 11×11: «Sea battle-aleks» 1.00, «Tyr (Martens)» +1.01.
- 9×9 бывают перекошены к осаждающим: «Ashton tablut» −5.26.

**Вывод для нас:** 11×11 по умолчанию делать **по Copenhagen-правилам** (сбаланс.
стандарт, без draw-fort). Tablut 9×9 — как «честный» средний режим. Brandubh 7×7
с расстановкой **cross** — быстрый/обучающий.

## 3. ИИ для тафла (✅ верифицировано) → рекомендации

Факты:
- **Фактор ветвления > 100** (у шахмат/шашек 10–20) → глубокий перебор тяжёл,
  особенно на 11×11. Сложность пространства состояний: от уровня шахмат до
  «между шахматами и го».
- **OpenTafl** (эталон): alpha-beta + iterative deepening + **transposition
  tables** + **killer move** + **history heuristic** + **move ordering** +
  ручная позиционная оценка. На 11×11 достаёт лишь **глубину 5–6** (~30 c); 9×9
  перебираем почти насквозь. **Оценочная функция — главный рычаг и слабое место.**
- **MCTS** в тактичных «ловушечных» играх (а тафл такой) играет слабо; помогает
  **гибрид MCTS+minimax** (мелкий minimax как приоры узлов/роллауты), но он
  чувствителен к ветвлению — нужен хороший move ordering / k-best / killer moves.
- **playtafl.org** гоняет клиентский **wasm-движок «taflzero»** (AlphaZero-стиль,
  Rust) + MCTS «Monty».
- Нейросеть-оценка через эволюцию (ES) → **~80% winrate** за 300 поколений;
  заодно инструмент **проверки баланса** рулсетов.

**Рекомендация для GDScript-ИИ (честный, многоуровневый):**
1. Базис — **alpha-beta + iterative deepening + transposition table (Zobrist) +
   move ordering** (killer/history). Это переносимо из OpenTafl.
2. **Оценочная функция** (главное): материал; мобильность короля и **дистанция
   короля до угла/кромки**; контроль углов/кромок осаждающими; прогресс окружения;
   «целостность» защитной стены. На неё делаем упор.
3. **Уровни сложности — честно**, не читом: лимит глубины/времени + сила оценки.
   9×9 — глубже; 11×11 — мельче глубина, но сильнее оценка/ordering.
4. Долгосрочно — опционально гибрид MCTS+minimax или wasm-движок для «эксперта».

## 5. Yandex SDK и мультиплеер (✅ верифицировано) — ВАЖНО

**Вывод: настоящего realtime-PvP/матчмейкинга у Yandex SDK НЕТ.** Что есть:
- **«Асинхронный мультиплеер»** = таймлайн/«призраки»: SDK записывает события с
  таймстампами и **проигрывает действия соперника-призрака**, когда играет другой
  игрок. Это **реплей**, не realtime. (`multiplayer_sessions_push`, meta1/meta2
  для матчинга — видно в defold-yagames.)
- **player.setData/getData** — KV на игрока, **200 КБ**, лимит ~100/5мин;
  `setStats/getStats/incrementStats` (10 КБ, 60/мин); авторизация, server-time,
  remote config, рейтинг, IAP, реклама.
- Хранилище **только на игрока** — общего «комнатного» состояния нет → честный
  realtime требует **внешнего бэкенда**.

**Пути для мультиплеера (реалистично):**
1. **Phase 1:** сильный SP vs ИИ + **локальные 2 игрока** (дёшево, сразу).
2. **Phase 2 (без бэкенда, в духе Yandex):** **асинхронный по ходам** через
   **код матча / private room** (как secret-code у Hnefatafl Online) — состояние
   матча в player data или передачей кода; и/или «призраки»/таймлайн; лидерборды.
3. **Phase 3 (опц.):** настоящий realtime/матчмейкинг через **внешний бэкенд** —
   **Nakama** (open-source Apache2, **клиент для Godot 4 на GDScript**, работает в
   HTML5 через WebSocket-адаптер, authoritative+relayed, матчмейкинг, тик до 30Гц)
   или Photon.

**Рекомендация:** не обещать realtime на старте. Делать SP+local → async/code-PvP
(идеально ложится на Yandex) → realtime через Nakama только если оправдано.

## 6. UX / онбординг (✅ верифицировано) → как учить

- **Progressive disclosure**: по одной механике за раз — *движение → custodial-
  взятие → трон/углы → побег короля → shieldwall/fort*. Уровней раскрытия < 3.
- Учить **когда механика становится нужна**; минимум теории, максимум практики;
  **безопасная песочница** до «живой» игры.
- **«Glowing choice»** — подсветить единственный следующий ход/клетку; подсветка
  легальных ходов (модель Lichess «move coach»).
- **Мост через знакомое**: «ходит как ладья», «бьёт зажатием» → привязка к
  шахматам/шашкам.
- **Асимметрия → отдельные обучающие треки** за осаждающих и за короля; игроку
  нужно несколько партий, чтобы понять взаимодействие сторон.
- **Цифровой энфорсмент правил**: блокировать нелегальные ходы, авто-подсветка
  взятий/угроз — новичок учится безопасно.
- **Самооценка опыта** (новичок/знаком/ветеран) адаптирует туториал; **скип** для
  возвращающихся.
- **Ранние «победные состояния»**: первая значимая победа = конец онбординга;
  «убивает удержание **путаница, а не скука**» → дать почувствовать себя умным
  быстро; просьбы «позови друзей»/соц — **после** первой победы.
- «Невидимый» туториал в зоне потока; в напряжённые моменты — явные подсказки, в
  спокойные — тонкие. Модель «Learn»-раздела: уроки + разбор ошибок после партии
  (Chess.com/Duolingo) поверх опциональной подсветки (Lichess).
